## Overview

**HyDE** (Hypothetical Document Embeddings) is a search technique where you write what the ideal answer looks like, then embed that hypothetical document to find real documents that match.

Instead of asking a question, you write 50-100 words that describe the answer you're looking for.

## When to Use hyde

**Use hyde when:**
- The topic is nuanced or complex
- You know what a good answer looks like
- Keyword and semantic search aren't precise enough

**Examples:**
- Describing a technical architecture pattern
- Writing out a hypothetical code snippet
- Drafting a paragraph that captures the concept

**Don't use hyde when:**
- Simple keyword search would work
- You're not sure what the answer should contain

## Query Format

Write 50-100 words describing the ideal answer:

```json
{
  "type": "hyde",
  "query": "The rate limiter uses a token bucket algorithm. When a client exceeds 100 requests per minute, subsequent requests return HTTP 429 until the time window resets. This prevents abuse while allowing legitimate burst traffic."
}
```

## How It Works

1. **Embed Hypothetical Document** — Your passage is converted to a 384-dimensional vector
2. **Similarity Search** — Find chunks that are semantically similar to your hypothetical document
3. **Rank Results** — Documents are scored by their best matching chunk

The key insight: **documents that answer your question will be similar to your hypothetical answer**.

## Example: Architecture Pattern

**Query:**
```json
{
  "searches": [
    { "type": "lex", "query": "circuit breaker" },
    { 
      "type": "hyde",
      "query": "The circuit breaker pattern prevents cascading failures by monitoring error rates. After a threshold of failures, the circuit opens and rejects requests immediately without calling the downstream service. After a timeout, it enters a half-open state to test recovery."
    }
  ]
}
```

This finds documents about circuit breakers even if they use different terminology or implementation details.

## Example: Code Pattern

**Query:**
```json
{
  "searches": [
    {
      "type": "hyde",
      "query": "async function retry<T>(fn: () => Promise<T>, attempts: number): Promise<T> { for (let i = 0; i < attempts; i++) { try { return await fn(); } catch (err) { if (i === attempts - 1) throw err; await sleep(1000); } } }"
    }
  ]
}
```

Finds retry logic implementations even if the exact syntax differs.

## Combining with lex and vec

Best practice: **combine all three** for maximum recall.

```json
{
  "searches": [
    { "type": "lex", "query": "\"connection pool\" timeout" },
    { "type": "vec", "query": "why do database connections time out under load?" },
    { 
      "type": "hyde",
      "query": "Connection pool exhaustion occurs when all connections are in use and new requests must wait. This typically happens under high concurrency when queries run longer than expected, causing the pool to fill up and subsequent requests to timeout."
    }
  ],
  "minScore": 0.5
}
```

- **lex** catches exact terminology
- **vec** finds conceptually similar content
- **hyde** captures nuanced descriptions

Results are combined via Reciprocal Rank Fusion (RRF). First sub-query gets 2× weight.

## Tips

### Write Specific Hypothetical Passages

**Good:**
```json
{
  "type": "hyde",
  "query": "The token bucket algorithm maintains a bucket with a maximum capacity. Tokens are added at a constant rate. Each request consumes one token. If no tokens are available, the request is rejected."
}
```

**Too Vague:**
```json
{
  "type": "hyde",
  "query": "Rate limiting works by limiting requests."
}
```

### Length Matters

- **50-100 words** is ideal
- Too short: insufficient signal
- Too long: dilutes the embedding

### Match the Content Type

If searching code:
```json
{ "type": "hyde", "query": "function handleError(err: Error) { logger.error(err); return { success: false }; }" }
```

If searching documentation:
```json
{ "type": "hyde", "query": "Error handling should log the error and return a structured response with success: false and an error message." }
```

## Limitations

- Requires embeddings (run `qmd embed`)
- Effectiveness depends on how well you can articulate the hypothetical answer
- No support for negation (use `lex` for that)

## CLI Equivalent

There's no direct CLI equivalent for pure HyDE search. Use the `query` tool via MCP, or use `qmd query` which may include HyDE as part of automatic expansion.
